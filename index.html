Problem

Probabilistic inference is not deterministic execution authorization.

When upstream sources conflict — sensors disagree, models hallucinate, channels go Byzantine — probabilistic guardrails average conflicting signals into a single output. That output can trigger unsafe execution. Most stacks gate content, not execution.

Solution

MGOS enforces a hard boundary between inference and execution.

The kernel sits at the last point before actuation. It does not modify actions — it authorizes, blocks, or stops them (ALLOW / BLOCK / STOP). Contradiction reduces available actions to zero: BLOCK, plus a hash-verified receipt explaining why. MGOS blocks execution on conflict — never averages.

Formal Properties (testable, falsifiable)
Property Falsifier / Test
Termination Bounded loops under input caps; falsified if any input causes non-termination. Test: max-time ceiling in evidence bundle.
Fail-safe Conflict or no-majority ⇒ BLOCK / STOP; falsified if any conflict yields ALLOW. Test: injection suite in bundle.
Determinism Same input ⇒ same output + hash under pinned environment; falsified if replay produces different hash. Test: determinism replay in bundle.
Receipts Hash-chain detects edits; falsified if modified receipt passes verification. Note: tamper-evident (detects edits), not tamper-resistant (no external anchoring by default). Receipt verification = hash-chain + bundle integrity (MANIFEST/SHA) — no external timestamp or blockchain. Test: tamper test in bundle.
Composition Wrapper integration does not weaken fail-safe; falsified if any wrapper path yields ALLOW on conflict. Test: composition smoke test in bundle (wrapper ⊇ kernel guarantees).
Explicitly Out of Scope / Threat Model
// HARDWARE
Hardware faults
Software fail-safes do not mitigate cosmic rays or silicon errors. Hardware redundancy (watchdogs, lockstep, diverse redundancy) remains a deployment obligation.
// HOST
Host compromise
MGOS assumes an uncompromised host OS (Ring-0 / root integrity). Host hardening is a deployment obligation.
// ALIGNMENT
AI alignment / truth evaluation
MGOS enforces bitwise execution authorization. It does not evaluate semantic truth or ethical alignment.
// CROSS-PLATFORM
Cross-platform WCET
WCET bounds are reported for the pinned evaluation environment (A1–A2). Cross-platform WCET is out of scope; operators must re-certify on target hardware.
Architectural invariants
Fail-open is architecturally forbidden — the gate defaults to STOP on any unresolvable state.
DoS against MGOS process degrades to STOP (safe), assuming deployment treats MGOS unavailability as STOP.
Receipts are hash-verified and tamper-evident; compromising them requires access to the host environment (out of MGOS scope).

Integration (minimal contract)

Minimal contract for integration partners.

MGOS sits between your inference stack and your execution layer. It receives a decision frame, returns ALLOW, BLOCK, or STOP, and emits a receipt.

// IO Contract
Input: decision_frame { channels[], policy_ref, timestamp }
Output: { decision: ALLOW | BLOCK | STOP, state_id, flags, receipt_hash }
Where: Layer 4 — between inference (L0–L3) and execution (L5)
// Minimal integration (pseudocode)

frame = build_decision_frame(channels, policy)
resp = mgos.authorize(frame)

if resp.decision != ALLOW:
    log(resp.receipt_hash, resp.flags)
    abort()

// only reachable on ALLOW
execute(action, receipt=resp.receipt_hash)

FAQ (engineering objections)
What is MGOS, in one sentence?
A deterministic runtime authorization kernel (veto layer) between inference and execution that outputs ALLOW / BLOCK / STOP and emits a hash-verified receipt.
How is this different from a rules engine?
Rules engines define business logic. MGOS enforces an execution boundary with hard properties: determinism, bounded time, fail-safe on contradiction, and replayable evidence artifacts.
What is the latency overhead?
Bounded and measured under explicit input caps. Exact figures are provided in the evidence bundle (perf microbench + logs). Measurement conditions are pinned and declared.
What happens if MGOS crashes?
Deployment must treat MGOS as authoritative: if the guard is unavailable, the system must default to fail-safe (BLOCK/STOP). Hardware faults can bypass software fail-safes. Hardware redundancy (watchdogs, lockstep, diverse redundancy) remains a deployment obligation.
Do you support OPA/Rego or custom policy languages?
MGOS is policy-agnostic. Policy configuration is external to the kernel. Integration can map OPA/Rego decisions into the MGOS decision_frame contract.
Is MGOS "AI safety" / alignment?
No. MGOS does not evaluate truth or align models. It enforces deterministic runtime authorization and produces audit evidence.
Does MGOS require network access or telemetry?
No. The kernel operates entirely offline. Verification and evidence bundles are self-contained.
What platforms do you support?
Desktop evaluation app: macOS (current), Windows (planned). Kernel (CLI/SDK): macOS, Linux (build available under eval agreement), Docker packaging planned.
